# 团队协作与规划规则

## 规划文件结构

```
docs/planning/
├── PROJECT_STATUS.md    # 当前进度上下文（第一步必读）
├── PROJECT_PLAN.md      # 总计划（里程碑、阶段目标）
└── goals/
    └── 001-*.md         # 单目标细案
```

## 规划文件维护规则

**何时更新规划文件：** 只有当任务改变了以下内容之一时才同步更新对应规划文件：
- 目标状态（完成/变更/废弃）
- 下一步计划
- 已识别的风险
- 验证结果
- 团队决策

**不要把 PROJECT_STATUS.md 当作每次对话都必须追加的工作日志。** 持续在末尾追加内容会导致状态过时且难以阅读。

## 变更工作流

1. 开始任何代码或文档变更前，先阅读 `docs/planning/PROJECT_STATUS.md`
2. 按需阅读 `docs/planning/PROJECT_PLAN.md` 和相关 `goals/*.md`
3. 完成变更后，如果触及"何时更新"中的任何一条，更新对应文件

## 渐进式披露

- `PROJECT_STATUS.md`：浓缩的当前状态。读了它就知该做什么。
- `PROJECT_PLAN.md`：总体蓝图。回答"这个项目分几步、为什么这样分"。
- `goals/*.md`：单目标的详细规格。回答"这一步具体怎么做、验收标准是什么"。

如果归档的旧计划与当前规划文件冲突，以当前规划文件为准。
